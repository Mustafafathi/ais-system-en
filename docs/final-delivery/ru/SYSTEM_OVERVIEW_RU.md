# Полное описание системы AIS V2

## 1. Назначение системы

AIS - это автоматизированная информационная система учёта посещаемости студентов. Система предназначена для [University Name Redacted] и заменяет ручные журналы, бумажные ведомости и разрозненные Excel-файлы единой цифровой моделью посещаемости.

Система решает следующие задачи:

- фиксация посещаемости вручную преподавателем;
- фиксация посещаемости по QR-коду;
- приём событий от СКУД;
- хранение и обработка оправдательных документов;
- уведомления студентам и ответственным сотрудникам;
- отчёты по студентам, группам, преподавателям, периодам и произвольным шаблонам;
- централизованное управление пользователями, ролями, справочниками и настройками;
- offline-режим для устойчивой работы при нестабильной сети;
- интеграция через CSV с 1C без прямого сетевого подключения.

## 3. Основные пользовательские сценарии

### 3.1 Авторизация

1. Пользователь открывает `login/index.php`.
2. Форма отправляет запрос к `api.php` с action `Авторизация`.
3. SQL Server проверяет логин, пароль, блокировку, роль и состояние пользователя.
4. PHP сохраняет `session_id` и `token`.
5. `index.php` перенаправляет пользователя на рабочее место его роли.

### 3.2 Проверка доступа к защищённой странице

1. Страница подключает `includes/auth_check.php`.
2. Проверяется наличие `$_SESSION['session_id']` и `$_SESSION['token']`.
3. Выполняется `ПроверитьСессию`.
4. Загружается пользователь через `ПолучитьПользователяПоID`.
5. Загружается навигация через `ПолучитьНавигациюПользователя`.
6. При необходимости выполняется `ПроверитьДоступКСтранице`.
7. Если доступ запрещён, возвращается `403`.

### 3.3 Ручная отметка посещаемости

1. Преподаватель открывает журнал посещаемости.
2. UI запрашивает занятия через `ПолучитьЗанятияПоДате`.
3. Для выбранного занятия загружается список студентов через `ПолучитьПосещаемостьПоЗанятию`.
4. При изменении статуса вызывается `ОтметитьПосещаемость`.
5. SQL Server проверяет права, занятие, статус, дубли и аудит.
6. UI обновляет строку журнала.

### 3.4 QR-посещаемость

1. Преподаватель открывает `teacher/qr-generator.php`.
2. Выбирает занятие из расписания.
3. Если фактическое занятие ещё не создано, вызывается `СоздатьЗанятие`.
4. Вызывается `СгенерироватьQRДляЗанятия`.
5. В интерфейсе отображается QR-код через локальную библиотеку `assets/vendor/qrcode.min.js`.
6. Студент открывает `student/qr-scanner.php`.
7. Сканер использует локальный адаптер `assets/vendor/html5-qrcode.min.js`.
8. После чтения кода вызывается `ПроверитьQRИОтметить`.
9. SQL Server проверяет активность QR-сессии, срок действия, студента, занятие и повторное сканирование.
10. Результат возвращается студенту.

### 3.5 Offline-режим

1. Клиентский JS отслеживает `navigator.onLine`.
2. При отсутствии сети поддерживаемая операция сохраняется в IndexedDB `AIS_OfflineQueue`.
3. Если IndexedDB недоступен, используется `localStorage`.
4. Для запроса формируется `idempotency_key`.
5. При восстановлении сети очередь отправляется в `offline-handler.php`.
6. Сервер повторно вызывает `api.php`, передавая `Idempotency-Key`.
7. `api.php` хранит краткоживущий результат в `runtime/idempotency`, чтобы не выполнить одно действие дважды.

### 3.6 Обоснования отсутствия

1. Студент открывает форму обоснования.
2. Выбирает занятие, причину и прикладывает файл, если требуется.
3. Создаётся запись через `СоздатьОбоснование`.
4. Куратор или преподаватель открывает список обоснований.
5. Решение выполняется через `ИзменитьСтатусОбоснования`.
6. SQL Server меняет статус, фиксирует решение, обновляет связанные данные посещаемости и журналирует операцию.

### 3.7 Отчёты

Система поддерживает отчёты:

- по группе;
- по студенту;
- по преподавателю;
- по дням;
- по неделям;
- по месяцам;
- произвольные отчёты;
- экспорт посещаемости в CSV;
- плановые отчёты через SQL Server Agent.

Основные процедуры: `СформироватьОтчетПоГруппе`, `СформироватьОтчетПоСтуденту`, `СформироватьОтчетПоПреподавателю`, `СформироватьОтчетПоДням`, `СформироватьОтчетПоНеделям`, `СформироватьОтчетПоМесяцам`, `СформироватьПроизвольныйОтчет`, `ЭкспортПосещаемостиВCSV`, `ВыполнитьПлановыйОтчет`.

## 5. Границы ответственности

| Слой | Разрешено | Запрещено |
| --- | --- | --- |
| UI | Отображение, сбор ввода, вызов API, локальная очередь offline. | Прямой доступ к БД, бизнес-расчёты, обход ролей. |
| PHP | Транспорт, сессии, чтение JSON/multipart, вызов SP, JSON-ответ. | Дублирование бизнес-правил, прямые SQL-операции из UI-данных. |
| SQL Server | Бизнес-логика, права, проверки, расчёты, аудит, интеграционная обработка. | Хранение web-секретов в коде клиента. |
| Интеграции | Проверка подписи, нормализация входа, вызов SP. | Принятие неподписанных или неразрешённых запросов. |
