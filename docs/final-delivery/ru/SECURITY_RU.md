# Безопасность AIS V2

## 1. Цели безопасности

Система должна защищать:

- персональные данные студентов и сотрудников;
- статусы посещаемости;
- основания отсутствия и вложения;
- учётные записи и сессии;
- интеграционные каналы SKUD и CSV;
- административные операции;
- журналы аудита и резервные копии.

## 2. Базовая модель угроз

| Угроза | Защита |
| --- | --- |
| SQL Injection | UI не формирует SQL; `api.php` вызывает только проверенные хранимые процедуры, параметры передаются через `sqlsrv_prepare`. |
| Обход роли | Проверка сессии и доступа к странице через БД. |
| Повтор offline-запроса | `Idempotency-Key` и файловый кэш `runtime/idempotency`. |
| Подмена SKUD-события | IP allowlist, HMAC, timestamp, nonce. |
| Просмотр внутренних файлов | `.htaccess` блокирует `includes/`, `Database/`, `integration/common/`, `runtime/`. |
| Утечка секретов | Пароли и секреты должны задаваться через переменные окружения. |
| XSS | UI должен экранировать динамический вывод; backend возвращает JSON. |
| CSRF | Для state-changing endpoints требуется session/token; рекомендуем усилить CSRF-token для форм. |
| Brute force | Логика блокировки и попыток входа должна находиться в `Авторизация`/БД. |

## 3. Аутентификация

Главные сущности:

- `Пользователь`;
- `Сессия_Пользователя`;
- `Авторизация`;
- `ПроверитьСессию`;
- `ЗавершитьСессию`.

Поток:

1. Пользователь вводит логин и пароль.
2. `api.php` вызывает `Авторизация`.
3. SQL Server проверяет hash/salt и состояние аккаунта.
4. Создаётся session/token.
5. PHP session хранит `session_id`, `token`, role/user ids.

В локальной поставке допускается демонстрационный fallback в `config.php` для запуска на стенде XAMPP. Для промышленной эксплуатации пароль БД должен передаваться через `AIS_DB_PASSWORD`, а fallback должен быть удалён или заменён безопасной конфигурацией.

## 4. Авторизация и доступы

Уровни контроля:

1. PHP route guard: `includes/auth_check.php`.
2. Проверка сессии: `ПроверитьСессию`.
3. Динамическая навигация: `ПолучитьНавигациюПользователя`.
4. Проверка страницы: `ПроверитьДоступКСтранице`.
5. Разрешения ролей: `Разрешения_Ролей`.
6. Процедурные проверки внутри SQL Server.

Важно: скрытие кнопки в UI не является достаточной защитой. Окончательное решение должно быть в БД.

## 5. Защита API

`api.php` выполняет:

- чтение тела запроса;
- проверку JSON;
- проверку имени `action` регулярным выражением;
- поиск процедуры в `INFORMATION_SCHEMA.ROUTINES`;
- чтение параметров из `INFORMATION_SCHEMA.PARAMETERS`;
- безопасное связывание параметров;
- проверку session/token для защищённых actions;
- единый JSON-ответ;
- idempotency для повторяемых запросов.

Недопустимо:

- добавлять прямой SQL на основе данных пользователя;
- принимать имя схемы от клиента;
- расширять список публичных процедур без security-review;
- возвращать raw stack trace в production.

## 6. Защита SKUD

Файл: `integration/skud/event.php`.

Механизмы:

- `AIS_SKUD_SECRET` обязателен;
- IP должен входить в `AIS_INTEGRATION_ALLOWLIST`;
- обязательные заголовки: `X-SKUD-Signature`, `X-SKUD-Timestamp`, `X-SKUD-Nonce`;
- HMAC рассчитывается по строке `timestamp.nonce.rawBody`;
- timestamp должен быть свежим;
- nonce защищает от replay;
- дубли nonce возвращаются как duplicate и не выполняются повторно.

## 7. Защита health endpoint

Файл: `integration/system/health.php`.

В V2 endpoint требует `AIS_HEALTH_SECRET`. Без секрета возвращается `503`, при неверном `X-System-Secret` возвращается `401`.

## 8. Защита файловой системы

`.htaccess`:

- запрещает доступ к `config.php`, `.htaccess`, `.git`, `.env`;
- запрещает прямой доступ к `includes/`;
- запрещает прямой доступ к `Database/`;
- запрещает прямой доступ к `integration/common/`;
- запрещает прямой доступ к `runtime/`;
- отключает listing директорий.

Права записи нужны только для:

- `runtime/sessions`;
- `runtime/idempotency`;
- `runtime/tmp`;
- `uploads`.

## 9. Управление секретами

Обязательные переменные:

| Переменная | Назначение |
| --- | --- |
| `AIS_DB_PASSWORD` | Пароль SQL-пользователя. |
| `AIS_SKUD_SECRET` | HMAC-секрет SKUD. |
| `AIS_HEALTH_SECRET` | Секрет health endpoint. |

Дополнительные:

| Переменная | Назначение |
| --- | --- |
| `AIS_DB_HOST` | Хост SQL Server. |
| `AIS_DB_PORT` | Порт SQL Server, по умолчанию `15432`. |
| `AIS_DB_NAME` | Имя базы. |
| `AIS_DB_USER` | SQL-пользователь. |
| `AIS_SITE_URL` | Базовый URL. |
| `AIS_DB_ENCRYPT` | Шифрование SQL-соединения. |
| `AIS_DB_TRUST_SERVER_CERT` | Доверие сертификату SQL Server. |
| `AIS_INTEGRATION_ALLOWLIST` | IP-allowlist интеграций. |

## 10. Аудит

Аудит реализуется через:

- `Лог_Действий`;
- `Ошибки_Системы`;
- `integration_audit_log`;
- триггеры логирования;
- процедуры мониторинга.

Аудит должен фиксировать:

- вход/выход;
- изменения пользователей и ролей;
- изменение посещаемости;
- решения по обоснованиям;
- административные операции;
- интеграционные события;
- ошибки доступа.

## 11. Production checklist

- `AIS_DEBUG=false`.
- `AIS_DB_PASSWORD` задан вне кода для production-среды; демонстрационный fallback в `config.php` отключён перед вводом в эксплуатацию.
- `AIS_SKUD_SECRET` задан и не равен тестовому значению.
- `AIS_HEALTH_SECRET` задан.
- SQL Server port `15432` закрыт от внешней сети, кроме нужных хостов.
- Включён HTTPS.
- Папки runtime не доступны по HTTP.
- Папка `Database` не доступна по HTTP.
- Созданы резервные копии.
- Проверены роли и тестовые аккаунты.
- Проверены SKUD HMAC, CSV, Database Mail и SQL Agent, если они в scope.
