# Архитектура AIS V2

## 1. Архитектурный стиль

AIS V2 построен по принципу "толстой базы данных":

- SQL Server является владельцем бизнес-правил.
- PHP является тонким транспортным слоем.
- UI работает только через HTTP/API.
- Интеграции нормализуют входящие данные и также вызывают хранимые процедуры.

Такой подход выбран для консистентности, безопасности и управляемости: любое правило посещаемости, доступа или отчётности находится рядом с данными и не расходится между страницами.

## 2. Физическая схема развёртывания

| Компонент | Тип | Назначение |
| --- | --- | --- |
| Браузер пользователя | Client | Рабочие места ролей, QR, offline queue. |
| Apache/XAMPP | Web server | Отдача PHP, CSS, JS, изображений. |
| PHP 8.2 + sqlsrv | Application transport | `api.php`, сессии, интеграционные endpoints. |
| SQL Server 2022 | DBMS | Таблицы, процедуры, триггеры, аудит, отчёты. |
| SQL Server Agent | Job engine | Плановые отчёты и регламентные операции, если включены. |
| Database Mail | Mail subsystem | Восстановление пароля и доставка отчётов, если включены. |
| SKUD | External system | События прохода через турникеты/карты. |
| 1C | External ERP | CSV-обмен данными без прямого подключения. |

## 3. Слой UI

UI реализован на PHP-шаблонах, HTML, CSS и JavaScript без frontend-framework.

Основные элементы:

- `includes/header.php` подключает общий CSS/JS.
- `includes/nav.php` строит навигацию по роли.
- `assets/css/style.css` содержит общий дизайн.
- `assets/js/common.js` содержит `callAPI`, `callAPIOfflineSupport`, хранение токенов, offline queue.
- `assets/js/offline-queue.js` предоставляет оболочку для очереди.
- `assets/js/schedule-explorer.js` отображает расписание.
- `assets/js/attendance-journal.js` обслуживает журнал посещаемости.

UI не должен принимать окончательные решения о праве доступа или корректности статуса. Любая проверка UI является вспомогательной.

## 4. PHP thin API

Главный файл: `api.php`.

Основной алгоритм:

1. Прочитать JSON/multipart/GET.
2. Проверить наличие и формат `action`.
3. Нормализовать параметры.
4. Извлечь `Authorization: Bearer <token>` или token из payload.
5. Для защищённых процедур выполнить `ПроверитьСессию`.
6. Проверить существование процедуры в `INFORMATION_SCHEMA.ROUTINES`.
7. Получить параметры процедуры из `INFORMATION_SCHEMA.PARAMETERS`.
8. Связать входные и выходные параметры.
9. Выполнить `EXEC [dbo].[procedure] @param = ?`.
10. Вернуть JSON.

Отдельная логика:

- публичные процедуры: `Авторизация`, `ВосстановитьПароль`, `ПодтвердитьВосстановлениеПароля`, `ПроверитьСессию`;
- CSV import actions: `ИмпортГруппИзCSV`, `ИмпортСтудентовИзCSV`;
- idempotency для повторяемых offline-запросов.

## 5. Слой базы данных

SQL Server содержит:

- справочники;
- пользователей, роли и права;
- расписание и занятия;
- посещаемость;
- QR-сессии;
- события SKUD;
- обоснования отсутствия;
- уведомления;
- отчёты;
- аудит и ошибки;
- триггеры консистентности.

Хранимые процедуры являются внешним контрактом backend-слоя. PHP вызывает процедуры по имени `action`.

## 6. Интеграционный слой

Интеграции расположены в `integration/`:

- `integration/skud/event.php` - приём событий SKUD;
- `integration/csv/mapping.php` - связь API action и CSV-процедуры;
- `integration/csv/normalizers.php` - нормализация CSV;
- `integration/system/health.php` - системная проверка состояния;
- `integration/common/*` - общие функции ошибок, аудита и вызова процедур.

## 7. Runtime-хранилища

| Путь | Назначение |
| --- | --- |
| `runtime/sessions` | fallback-хранилище PHP-сессий, если системный путь недоступен. |
| `runtime/idempotency` | файловый кэш idempotency-ответов для offline replay. |
| `runtime/tmp` | временные файлы приложения. |
| `uploads` | вложения обоснований и другие загружаемые файлы, если включены. |

Содержимое runtime не входит в исходные данные поставки и не должно переноситься между средами как бизнес-данные.

## 8. Контракты между слоями

### UI -> API

```json
{
  "action": "ПроверитьСессию",
  "params": {},
  "session_id": "...",
  "token": "..."
}
```

### API -> DB

```sql
EXEC [dbo].[ПроверитьСессию] @Сессия_ID = ?, @Токен = ?
```

### API -> UI

```json
{
  "success": true,
  "data": [],
  "message": ""
}
```

## 9. Масштабирование и сопровождение

Система ориентирована на размер порядка 3 000-5 000 студентов и 200-300 преподавателей. Для эксплуатации важны:

- индексы по пользователям, группам, расписанию, занятиям, посещаемости;
- контроль роста журналов;
- регламентное обслуживание через процедуры `ЕжедневноеОбслуживаниеСистемы`, `ОбслуживаниеИндексов`, `ОчиститьСтарыеЛоги`;
- мониторинг SQL Server, Apache и PHP;
- регулярное резервное копирование.
