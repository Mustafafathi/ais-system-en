# Тестирование и приёмка AIS V2

## 1. Цели тестирования

Тестирование должно подтвердить:

- чистоту поставки V2;
- синтаксическую корректность PHP;
- работоспособность подключения к SQL Server;
- корректность API-контракта;
- соответствие ролей и доступов;
- корректность посещаемости, QR и offline;
- безопасность интеграций;
- качество UI на разных экранах;
- готовность документации к передаче.

## 2. Уже выполненные проверки пакета V2

## 4. API smoke

### 4.1 Проверка сессии с неверными данными

Ожидается controlled error, не PHP fatal:

```http
POST /ais-system-ru/api.php
Content-Type: application/json

{
  "action": "ПроверитьСессию",
  "params": {
    "session_id": "x",
    "token": "y"
  }
}
```

### 4.2 Авторизация

1. Отправить `Авторизация` с валидным логином/паролем.
2. Ожидать `success=true`.
3. Получить `session_id`, `token`, роль.
4. Проверить `ПроверитьСессию`.

### 4.3 Защищённая операция без token

1. Вызвать защищённый action без token.
2. Ожидать `401`.
3. Убедиться, что процедура не выполнена.

## 5. DB rollback-тесты

Каждый destructive/create/update сценарий должен выполняться в транзакции:

```sql
BEGIN TRANSACTION;
EXEC dbo.<procedure> ...;
-- assertions
ROLLBACK TRANSACTION;
-- verify no persistent change
```

Критические сценарии:

- создание пользователя;
- создание студента;
- создание преподавателя;
- изменение роли;
- изменение разрешения;
- создание расписания;
- создание занятия;
- ручная отметка посещаемости;
- создание QR-сессии;
- повторное QR-сканирование;
- создание и утверждение обоснования;
- update настройки;
- административная операция.

## 7. Security-тесты

| Тест | Ожидание |
| --- | --- |
| Вызов `Database/Улучшенная.sql` по HTTP | Доступ запрещён. |
| Вызов `includes/auth_check.php` напрямую | Доступ запрещён или не раскрывает данные. |
| Вызов `integration/common/*` напрямую | Доступ запрещён. |
| SKUD без HMAC | `401`. |
| SKUD с неверным IP | `403`. |
| SKUD с повторным nonce | duplicate без повторного действия. |
| Health без секрета | `401` или `503`, если секрет не настроен. |
| Protected API без token | `401`. |
| Неверный action name | `400`. |

## 8. Responsive и accessibility

Проверить viewport:

- 375 px;
- 768 px;
- 1024 px;
- 1440 px.

Проверить:

- отсутствие горизонтального overflow;
- читаемость таблиц;
- focus states;
- keyboard navigation;
- контраст;
- доступные подписи форм;
- отсутствие перекрытия текста.

## 9. Критерии приёмки

Система может быть принята как готовая к эксплуатации, если:

- все mandatory checks имеют PASS;
- все критические риски закрыты;
- секреты вынесены в окружение;
- SQL Server, Apache, Database Mail и SQL Agent проверены в scope;
- роли работают согласно матрице доступа;
- посещаемость, QR, offline и обоснования проходят E2E;
- интеграции SKUD/CSV подтверждены, если используются;
- оператор получил документацию и инструкции восстановления.
